# Enterprise Requirement Doc

## Use This Skill When

- 用户要求"生成需求文档""写需求说明书""输出功能需求文档""复刻一份企业级需求文档"。
- 用户提供了 `docx`、`pdf`、`md`、`txt`、图片、会议纪要、建设方案、功能说明书、界面原型、已有需求文档、模板规范等原材料。
- 用户要求先形成结构化中间稿，再输出正式需求文档。
- 用户要求对生成结果做 AI 自评，确保准确、可行、不偏离原材料。

## Default Deliverables

默认同时输出以下文件；若用户指定命名或目录，按用户要求覆盖：

- `01-需求新增优化改造点.md` - 中间稿（含业务价值量化、MoSCoW 优先级、兼容性分析）
- `02-系统功能需求说明书.md` - Markdown 格式正式文档（含 AC 验收标准、数据追踪埋点、Unhappy Path）
- `02-系统功能需求说明书.docx` - **Word 格式正式文档（必选）**
- `03-AI自评报告.md` - AI 自评报告（含 AC 完整性、埋点完整性检查）
- `04-AI局限性声明.md` - AI 生成内容的局限性和人工复核清单

**重要**：以上 5 个文件必须全部生成，特别是 DOCX 文件不得遗漏。

**Pro 版增强特性**：
- ✅ **AC 验收标准**：每个功能点采用 Given-When-Then 格式
- ✅ **Unhappy Path**：强制要求至少 2 个异常场景
- ✅ **数据追踪埋点**：完整的事件列表和属性定义
- ✅ **业务价值量化**：每个需求点都有可量化的价值说明
- ✅ **兼容性分析**：改造点的历史兼容性和迁移方案
- ✅ **苏格拉底式追问**：防御性边界问题清单

## Hard Rules

- **禁止简单提取**：不得将原材料内容直接复制粘贴到文档中，必须经过分析、归纳、展开后输出。
- 先抽取事实，再扩写文档，禁止直接"脑补成文"或"简单搬运"。
- 明确区分 `事实`、`推断`、`待确认` 三类信息。
- 如缺少关键内容，必须先提问，不得用想当然内容补全关键业务规则。
- 公司模板优先于默认模板；公司模板未覆盖的部分再用默认模板补齐。
- 输出应以中文为默认语言；技术术语保留原文。
- 中间稿必须显式整理 `新增点`、`优化点`、`改造点`，并标注来源依据。
- 自评发现重大冲突、关键缺口、明显不可行点时，先修正再交付。
- **Phase 3.5（深度分析）必须完整执行，不得跳过或简化**。

## Workflow

按以下顺序执行，不要跳过中间稿和自评。

### Phase 1: Ingest Materials

1. 读取用户需求描述与全部原材料。
2. 按材料类型分类：
   - `docx`：优先运行 `python scripts/extract_docx.py <input.docx>` 提取正文和表格（需要安装 `pip install python-docx mammoth`）。
   - `doc`：运行 `python scripts/extract_docx.py <input.doc>` 提取正文（使用 mammoth 库，需要安装 `pip install mammoth`）。
   - `pdf`：优先直接读取文本内容；若为扫描件（无文字图层），使用 OCR 工具提取文字后再处理。
     - 文本型 PDF：直接读取，可提取表格（推荐 `pdfplumber`）
     - 扫描件 PDF：先 OCR 识别文字，再按文本处理
   - `md/txt`：直接读取。
   - `png/jpg/jpeg/webp`：提取页面区域、字段、按钮、列表、状态、说明文案。
3. 若材料存在冲突，记录冲突点，不要私自裁决。

补充规则见 [docs/material-analysis-guide.md](docs/material-analysis-guide.md)。

### Phase 2: Build Fact Base

1. 按以下维度建立事实底稿：
   - 背景与痛点
   - 建设目标
   - 范围与边界
   - 角色与职责
   - 核心对象、状态、状态流转
   - 功能模块
   - 页面与交互
   - 业务规则
   - 接口与数据
   - 非功能要求
   - 风险、依赖、约束、待确认事项
2. 对每条信息标记来源材料。
3. 执行关键缺口检查：
   - 业务目标
   - 功能模块清单
   - 核心对象/状态流转
   - 角色权限
   - 范围边界
4. 若缺失关键项，优先只问 1-3 个最关键问题。

分析口径见 [docs/material-analysis-guide.md](docs/material-analysis-guide.md)。

### Phase 3: Generate Delta Markdown

1. 基于事实底稿整理 `新增点`、`优化点`、`改造点`。
2. 每一点都必须包含：
   - 来源材料
   - 原文依据或归纳依据
   - 需求描述
   - 影响范围
   - 规则/约束
   - 落地建议
   - 优先级/必要性
   - 不确定项
3. 输出到 `01-需求新增优化改造点.md`。

模板见 [templates/delta-template.md](templates/delta-template.md)。

### Phase 3.5: Deep Analysis（深度分析）⚡ 关键步骤

**这是解决"内容深度一般"问题的核心步骤。不要跳过！**

在生成正式文档前，必须像资深需求分析师一样进行深度思考：

#### 3.5.1 追问澄清（Clarification）

对每个功能点追问以下问题：

| 问题类型 | 具体问题 | 输出要求 |
|---------|---------|---------|
| 业务价值 | 这个功能解决了什么痛点？如果不做会有什么影响？ | 必须用数据量化价值 |
| 目标用户 | 谁在什么场景下使用？用户当前的 workaround 是什么？ | 明确角色和使用频率 |
| 成功指标 | 怎么衡量这个功能的成功？可量化的指标是什么？ | 明确的 KPI |
| 替代方案 | 有没有其他实现方式？为什么选当前方案？ | 对比分析的结论 |

#### 3.5.2 场景拆解（Scenario Breakdown）

每个功能必须拆解以下场景：

- **主流程（Happy Path）**：正常情况下的完整操作流程
- **异常流（Exception Flow）**：系统出错、网络中断、数据异常时的处理
- **边界 case（Edge Cases）**：
  - 极端数据（空值、超大值、特殊字符）
  - 极端并发（高并发、并发冲突）
  - 极端操作（快速重复点击、超时、断网重连）

#### 3.5.3 冲突检测（Conflict Detection）

检查以下内容是否存在冲突：

- [ ] 不同原材料之间的业务规则是否一致
- [ ] 新需求与历史需求是否存在逻辑冲突
- [ ] 功能之间是否存在依赖关系（A 功能依赖 B 功能）
- [ ] 数据定义是否统一（同一字段在不同地方定义是否一致）

**若发现冲突**：
1. 在 `01-需求新增优化改造点.md` 中标注冲突点
2. 分析冲突原因
3. 提出倾向性建议
4. 等待用户裁决

#### 3.5.4 数据流分析（Data Flow Analysis）

梳理以下数据流转：

| 数据实体 | 字段定义 | 数据来源 | 数据去向 | 更新频率 |
|---------|---------|---------|---------|---------|
| 示例：订单 | 订单号、金额、状态 | 用户创建 | 支付系统、库存系统 | 实时 |

- 数据如何流入系统？
- 数据在系统内部如何流转？
- 数据如何流出系统？
- 数据存储在哪里？保留多久？

#### 3.5.5 风险预判（Risk Assessment）

识别以下风险：

| 风险类型 | 风险描述 | 影响程度 | 缓解措施 |
|---------|---------|---------|---------|
| 技术风险 | 技术实现难度、性能瓶颈 | 高/中/低 | 技术预研、POC |
| 业务风险 | 业务流程变更、用户接受度 | 高/中/低 | 灰度发布、培训 |
| 合规风险 | 数据安全、隐私保护、监管要求 | 高/中/低 | 法务评审、合规改造 |
| 依赖风险 | 第三方服务、外部接口 | 高/中/低 | fallback 方案 |

#### 3.5.6 优先级排序（Prioritization）

使用 MoSCoW 方法对需求排序：

- **Must have（必须有）**：不满足则系统无法上线
- **Should have（应该有）**：重要但不紧急，可以后续迭代
- **Could have（可以有）**：有则更好，没有也行
- **Won't have（不会有）**：明确不在当前版本

**输出要求**：
在 `01-需求新增优化改造点.md` 中补充以下内容：
1. 【深度分析】章节，包含上述 6 个维度的分析
2. 【待确认事项】清单，列出需要用户澄清的问题
3. 【风险提示】清单，列出识别的风险及建议

### ⚠️ 深度分析质量门控（进入 Phase 4 前必须通过）

**在进入 Phase 4 起草正式文档前，必须确认以下检查项：**

#### 门控检查项

- [ ] 每个功能模块都有业务价值说明（不能是"支持XX功能"一句话，必须量化：节省XX人天/提升XX%效率）
- [ ] 每个功能模块都有完整的主流程描述（步骤1→步骤2→...→完成）
- [ ] **每个功能模块都有至少 2 个 Unhappy Path（异常场景）**：
  - 业务异常：权限不足、数据不存在、状态不允许、校验失败
  - 系统异常：网络中断、超时、服务降级、并发冲突
- [ ] 每个功能模块都有边界 case 分析（空值、超大值、特殊字符、高并发等）
- [ ] 每个字段都有类型、长度、是否必填、含义说明
- [ ] 每个功能都有明确的角色权限定义
- [ ] 所有状态流转都有触发条件和结果状态
- [ ] 所有规则都有具体的校验逻辑或业务约束
- [ ] **新增/改造点都有兼容性分析**（历史数据、接口兼容、回滚方案）

#### Pro 版增强门控

- [ ] **每个功能点都有至少 3 个 AC（Acceptance Criteria）**：
  - 1 个主流程 AC（Happy Path）
  - 1 个异常流 AC（Unhappy Path - 业务异常）
  - 1 个边界流 AC（Unhappy Path - 系统异常/边界情况）
- [ ] **AC 采用 Given-When-Then 格式**，明确前置条件、操作、预期结果
- [ ] **核心功能都有数据追踪埋点定义**（事件名称、触发时机、关键属性）
- [ ] **改造点都有历史兼容性分析**（数据迁移、接口兼容、功能回滚）
- [ ] **所有 Must have 需求都有业务价值量化**（节省人力/时间/成本的具体数值）

**若门控检查未通过**：
1. 继续完善 Phase 3.5 深度分析，直到通过门控
2. 不得以"材料中没有"为由跳过深度展开
3. 确实缺少的信息，标注【待确认】而非留空

### Phase 4: Draft Formal Requirement Doc

**⚠️ 核心要求：每个功能模块必须完整展开，不得遗漏任何维度。Phase 3.5 深度分析的输出必须完整反映在 Phase 4 文档中。**

1. 先判断文档规模：
   - 轻量：<=3 个功能模块，集成少
   - 标准：4-10 个功能模块
   - 完整：>10 个模块或涉及复杂集成/多角色/AI/异步能力
2. 再判断能力类型：
   - A 类：传统业务功能
   - B 类：AI、异步、批处理、长耗时任务
3. 若用户提供公司模板，先融合模板结构，再补充缺失章节。
4. 按骨架先出大纲，再逐章成文。
5. 第四章系统功能必须优先使用统一模块结构，避免只写功能清单。
6. **每个功能模块必须包含以下完整内容**：
   - 【业务目标】：为什么要做，解决什么痛点（**必须量化**：预计节省XX人天/提升XX%效率/减少XX错误）
   - 【业务场景】：触发条件、目标用户、使用频率
   - 【业务流程】：主流程 + 异常流程 + 替代流程，完整操作步骤
   - **【AC 验收标准】**：至少 3 个 AC，采用 Given-When-Then 格式
   - **【Unhappy Path】**：至少 2 个异常场景（业务异常 + 系统异常）
   - 【字段说明】：字段名、类型、长度、必填、默认值、业务含义
   - 【校验规则】：输入校验、格式校验、业务规则、权限校验
   - 【异常处理】：错误代码、边界条件、超时处理、降级策略
   - 【接口需求】：若涉及系统间交互，提供接口签名和调用约定
   - 【数据流】：关键数据从输入到输出的完整路径
   - **【数据追踪埋点】**：核心操作的事件定义、触发时机、关键属性
   - 【优先级】：MoSCoW 标注（Must/Should/Could/Won't）
   - **【兼容性分析】**（仅改造点）：历史数据迁移、接口兼容、回滚方案
7. **禁止以下行为**：
   - 只列功能清单，不展开业务含义
   - 用"详见原型"敷衍字段说明和校验规则
   - 将原材料直接复制粘贴，不做分析和归纳
   - AC 写成"支持XX功能"的功能描述
   - Unhappy Path 只写"异常情况系统提示"而不具体说明

文档骨架见 [templates/document-template.md](templates/document-template.md)。
AC 写作规范见 [templates/ac-writing-guide.md](templates/ac-writing-guide.md)。
写作规范见 [docs/writing-guide.md](docs/writing-guide.md)。

### Phase 5: Produce Markdown And DOCX

**必须同时输出 Markdown 和 DOCX 文件，缺一不可。**

0. **输出前内容自检（必须逐项确认）**：
   - [ ] 每个功能模块都有业务目标、场景、流程描述
   - [ ] 每个字段都有类型/长度/必填/含义说明
   - [ ] 每个功能模块都有明确的校验规则和异常处理
   - [ ] 不存在"详见原型"、"待补充"等敷衍性表述
   - [ ] 不存在大段原材料直接复制粘贴
   - [ ] 所有【待确认】项都用灰色括号标注，而非留空
1. 先输出 `02-系统功能需求说明书.md`。
2. **立即调用脚本生成 DOCX**：
   ```bash
   python scripts/generate_docx.py --input 02-系统功能需求说明书.md --output 02-系统功能需求说明书.docx
   ```
3. 若脚本执行失败，必须手动使用 python-docx API 构建 DOCX：
   ```python
   from docx import Document
   # 参考 docx-generation-guide.md 中的代码模板
   ```
4. DOCX 生成规范：
   - 全局字体：微软雅黑 10.5pt
   - 多级标题：18/15/13/11.5pt，深色系配色
   - 表格：深蓝表头 + 白色字 + 交替行背景色
   - 封面页：标题居中 22pt + 元信息 12pt + 分页符
   - 文件名带版本号/日期，避免占用冲突
   - 生成前检查目标文件是否被 Word 占用
5. 详细的代码模板和配色方案见 [docs/docx-generation-guide.md](docs/docx-generation-guide.md)。
6. **重要**：不得跳过 DOCX 生成步骤，必须确保用户获得完整的 4 个交付物。

### Phase 6: Self Review

1. 基于正式文档和原材料执行 AI 自评。
2. 至少检查：
   - 准确性
   - 完整性
   - 可行性
   - 与原材料一致性
   - **AC 完整性（Pro 版新增）**
   - **数据追踪埋点完整性（Pro 版新增）**
   - **兼容性分析完整性（Pro 版新增）**
3. **深度专项检查（必须逐项执行）**：
   - [ ] 每个功能模块是否有业务目标、场景、流程（不是只有功能名称）
   - [ ] 字段说明是否包含类型/长度/必填/含义（不是只有字段名）
   - [ ] 校验规则是否具体可执行（不是只有"做校验"三个字）
   - [ ] 异常处理是否包含具体场景和策略（不是只有"异常时提示"）
   - [ ] 是否存在敷衍性表述（"详见原型"、"待补充"、"以实际为准"）
   - [ ] 是否存在大段原材料直接复制粘贴（不允许，必须分析后输出）
   - **Pro 版专项检查**：
   - [ ] **每个功能点是否都有至少 3 个 AC（Given-When-Then 格式）**
   - [ ] **AC 是否覆盖主流程、业务异常、系统异常三种场景**
   - [ ] **每个功能点是否都有至少 2 个 Unhappy Path**
   - [ ] **Unhappy Path 是否具体到错误代码和用户提示文案**
   - [ ] **核心功能是否都有数据追踪埋点定义**
   - [ ] **埋点定义是否包含事件名称、触发时机、关键属性**
   - [ ] **改造点是否都有兼容性分析**
   - [ ] **兼容性分析是否包含历史数据迁移方案**
   - [ ] **Must have 需求是否都有业务价值量化**
4. 输出 `03-AI自评报告.md`。
5. 若发现高风险问题，先回改正式文档，再给用户结果。
6. **自评未通过不得交付**：发现内容深度不足时，必须打回 Phase 4 重新展开。

自评规则见 [templates/self-review-checklist.md](templates/self-review-checklist.md)。

## Quality Bar

- 文档必须像企业级需求说明书，而不是摘要、脑图或产品清单。
- **内容深度要求**：每个功能模块必须展开到可执行程度，禁止仅列功能清单。
- **Pro 版质量要求**：
  - **AC 必须采用 Given-When-Then 格式，覆盖主流程 + 2 个 Unhappy Path**
  - **Unhappy Path 必须具体到错误代码、用户提示、系统处理逻辑**
  - **数据追踪埋点必须完整定义事件名称、触发时机、关键属性**
  - **改造点必须有兼容性分析和数据迁移方案**
  - **业务价值必须量化（节省XX人天/提升XX%效率/减少XX错误）**
- 每个复杂模块至少写到"业务目标/业务场景/业务流程/功能模块详情"。
- 页面级内容至少写清：入口、角色、界面区域、实现逻辑、字段说明、AC、Unhappy Path。
- 若涉及 AI/异步能力，补充交互过程、触发机制、验证指标。
- 状态、按钮、角色、字段名称全文统一。
- 不能编造接口名、状态值、权限规则、字段含义。

### 内容深度检查清单（生成文档前必须自检）

**每个功能模块必须包含：**
- [ ] 【业务目标】：为什么要做这个功能，解决了什么痛点（**必须量化**）
- [ ] 【业务场景】：在什么情况下会被触发，谁来用
- [ ] 【业务流程】：完整操作步骤，包含主流程和异常流
- [ ] **【AC 验收标准】：至少 3 个 AC（Given-When-Then 格式）**
- [ ] **【Unhappy Path】：至少 2 个异常场景（业务异常 + 系统异常）**
- [ ] 【字段说明】：每个字段的名称、类型、长度、是否必填、含义
- [ ] 【校验规则】：输入校验、业务规则校验、权限校验
- [ ] 【异常处理】：出错情况、边界情况、超时情况的处理方式
- [ ] **【数据追踪埋点】：核心操作的事件定义（仅核心功能）**
- [ ] **【兼容性分析】：历史数据迁移、接口兼容（仅改造点）**

**禁止出现以下内容：**
- ❌ "功能模块包括：用户管理、权限管理..."（无展开）
- ❌ "支持新增、编辑、删除、查询操作"（无细节）
- ❌ "系统会自动处理"（未说明处理逻辑）
- ❌ "待定"（关键业务规则不得为空）
- ❌ 原材料原文直接复制（必须经过分析归纳）
- ❌ **AC 写成"支持XX功能"的功能描述**
- ❌ **Unhappy Path 只写"异常情况系统提示"而不具体说明**
- ❌ **改造点不写兼容性分析**

## If The User Wants Fast Mode

- 仍然保留四个交付物，但可以减少中间确认轮次。
- 若材料充分，可连续执行；若关键要素缺失，仍必须先提问。
