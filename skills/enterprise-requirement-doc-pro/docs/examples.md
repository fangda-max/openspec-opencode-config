# 用法示例（Pro 版）

本文档提供 Pro 版的使用示例，包含 AC、Unhappy Path、埋点、兼容性分析等 Pro 版特性。

## 示例一：使用 Pro 版生成完整需求文档

### 用户输入

```text
请使用 Pro 版基于以下材料生成系统功能需求说明书：
- 会议纪要.docx
- 建设方案.docx
- 功能说明书.md
- 界面原型.pdf
- 公司需求模板.docx
```

### 期望执行
1. 读取全部材料并分类。
2. 抽取事实底稿，识别冲突与缺口。
3. 输出 `01-需求新增优化改造点.md`（含 MoSCoW、业务价值量化）。
4. 执行 Phase 3.5 深度分析（含苏格拉底式追问）。
5. 融合公司模板与默认模板。
6. 输出 `02-系统功能需求说明书.md`（含 AC、Unhappy Path、埋点、兼容性分析）。
7. 渲染 `02-系统功能需求说明书.docx`。
8. 输出 `03-AI自评报告.md`（含 Pro 版维度检查）。
9. 输出 `04-AI局限性声明.md`。

### 交付物

- `01-需求新增优化改造点.md` - 中间稿（含业务价值量化）
- `02-系统功能需求说明书.md` - 正式文档（含 AC、埋点）
- `02-系统功能需求说明书.docx` - Word 文档
- `03-AI自评报告.md` - 质量审查报告
- `04-AI局限性声明.md` - AI 局限性声明

## 示例二：AC 验收标准示例

### 功能点：用户登录

```markdown
【AC1：正常登录】
Given 用户已注册且账号状态正常
  And 用户处于登录页面
When 用户输入正确的用户名"zhangsan"和密码"Pass1234"
  And 点击【登录】按钮
Then 系统验证通过
  And 跳转至首页
  And 显示"登录成功"提示
  And 记录登录日志
  And 埋点事件"login_success"触发

【AC2：用户名不存在】
Given 用户处于登录页面
When 用户输入不存在的用户名"notexist"
  And 点击【登录】按钮
Then 系统拒绝登录
  And 提示"用户名或密码错误"
  And 错误代码：AUTH_USER_NOT_FOUND
  And 停留在登录页面
  And 清空密码输入框

【AC3：密码错误】
Given 用户处于登录页面
  And 用户"zhangsan"存在
When 用户输入用户名"zhangsan"和密码"wrongpass"
  And 点击【登录】按钮
Then 系统拒绝登录
  And 提示"用户名或密码错误"
  And 错误代码：AUTH_PASSWORD_ERROR
  And 记录失败日志
  And 连续错误 5 次后锁定账号 30 分钟

【AC4：网络超时】
Given 用户处于登录页面
  And 已输入正确的用户名和密码
When 用户点击【登录】按钮
  And 网络中断超过 30 秒
Then 系统停止登录流程
  And 提示"网络连接超时，请稍后重试"
  And 错误代码：NETWORK_TIMEOUT
  And 保留已输入的用户名
  And 显示【重试】按钮
```

## 示例三：Unhappy Path 详细示例

### 功能点：订单提交

```markdown
【Unhappy Path 详细说明】

**UP1：业务异常 - 库存不足**
- 触发条件：用户提交订单时，商品库存 < 购买数量
- 系统响应：拒绝提交，提示库存不足
- 错误代码：ORDER_INSUFFICIENT_STOCK
- 用户提示："商品【XXX】库存不足，当前仅剩 X 件"
- 页面反馈：高亮库存不足的商品行，显示实际库存
- 用户操作：可修改数量后重试，或删除该商品
- 埋点：事件"order_fail_insufficient_stock"触发

**UP2：业务异常 - 收货地址不在配送范围**
- 触发条件：用户选择的收货地址超出配送范围
- 系统响应：拒绝提交，提示配送限制
- 错误代码：ORDER_OUT_OF_DELIVERY_RANGE
- 用户提示："该收货地址不在配送范围内，请更换地址"
- 页面反馈：高亮收货地址选择区域
- 用户操作：可更换收货地址后重试
- 埋点：事件"order_fail_delivery_range"触发

**UP3：系统异常 - 订单服务超时**
- 触发条件：订单服务处理超过 10 秒未响应
- 系统响应：提示超时，数据状态不变
- 错误代码：ORDER_SERVICE_TIMEOUT
- 用户提示："系统繁忙，请稍后重试"
- 降级策略：本地保存订单草稿，用户可在"我的订单-草稿"中继续提交
- 重试机制：提供【重试】按钮，点击后重新提交
- 埋点：事件"order_timeout"触发

**UP4：系统异常 - 并发冲突（重复提交）**
- 触发条件：用户快速双击提交按钮
- 系统响应：幂等处理，只生成一个订单
- 错误代码：ORDER_DUPLICATE_SUBMIT
- 用户提示：无（正常跳转到订单详情页）
- 技术实现：前端防抖 + 后端幂等键（Idempotency Key）
- 埋点：事件"order_duplicate_submit"触发（仅记录，不提示用户）
```

## 示例四：数据追踪埋点示例

### 功能模块：订单管理

```markdown
【数据追踪埋点】

| 事件名称 | 事件类型 | 触发页面/操作 | 触发时机 | 业务含义 |
|---------|---------|--------------|---------|---------|
| order_page_view | 页面浏览 | 订单列表页 | 页面加载完成 | 统计订单功能使用率 |
| order_create_click | 按钮点击 | 订单列表页-新建订单按钮 | 用户点击新建订单 | 统计新建订单触发率 |
| order_submit_click | 按钮点击 | 订单创建页-提交按钮 | 用户点击提交 | 统计订单提交转化率 |
| order_create_success | 操作成功 | 订单创建页 | 订单成功创建 | 统计订单创建成功率 |
| order_create_fail | 操作失败 | 订单创建页 | 订单创建失败 | 统计订单创建失败率及原因 |
| order_detail_view | 页面浏览 | 订单详情页 | 页面加载完成 | 统计订单详情查看率 |
| order_export_click | 按钮点击 | 订单列表页-导出按钮 | 用户点击导出 | 统计订单导出使用率 |

【事件属性定义】

通用属性（所有事件都包含）：
| 属性名 | 类型 | 说明 | 示例 |
|-------|------|------|------|
| user_id | String | 用户唯一标识 | "U123456" |
| session_id | String | 会话标识 | "S789012" |
| timestamp | Long | 事件时间戳（毫秒） | 1712054400000 |
| device_type | String | 设备类型 | "PC" / "Mobile" |
| app_version | String | 应用版本 | "V2.1.0" |

事件特有属性：
| 事件名称 | 属性名 | 类型 | 说明 | 示例 |
|---------|-------|------|------|------|
| order_create_success | order_id | String | 订单号 | "ORD20240301001" |
| order_create_success | amount | Decimal | 订单金额（元） | 199.99 |
| order_create_success | duration_ms | Long | 创建耗时（毫秒） | 1500 |
| order_create_fail | error_code | String | 错误代码 | "INSUFFICIENT_STOCK" |

- 敏感数据脱敏：金额保留 2 位小数，不显示用户真实姓名
- 数据保留期限：≥ 6 个月
- 实时性要求：T+1 可查询
- 准确性要求：丢失率 < 1%
```

## 示例五：兼容性分析示例（改造项目）

### 改造点：订单状态流转改造

```markdown
【兼容性分析】

**改造背景**：
原订单状态：待支付 -> 已支付 -> 已发货 -> 已完成
新订单状态：待支付 -> 已支付 -> 待出库 -> 已发货 -> 已签收 -> 已完成

**历史数据迁移**：
- 数据量：历史订单约 500 万条
- 迁移方案：
  - 已发货状态的订单：映射为新状态"已发货"
  - 已完成状态的订单：映射为新状态"已完成"
  - 增量迁移：上线前全量迁移，上线后双写 1 个月
- 迁移时间：预计 4 小时，安排在周末凌晨 2-6 点
- 回滚方案：迁移前全量备份，如失败可回滚

**接口兼容性**：
- 接口变更：
  - 原接口 /api/order/status（返回旧状态值）
  - 新接口 /api/order/status/v2（返回新状态值）
- 向后兼容：保留原接口 3 个月，返回兼容状态值
- 调用方影响：
  - 内部系统 A：已改造，调用新接口
  - 外部系统 B：未改造，仍调用原接口（兼容期内）
- 废弃下线时间：原接口计划于 2024-06-01 下线

**功能回滚**：
- 回滚触发条件：新状态流转出现严重 Bug，影响业务流程
- 回滚范围：全量回滚（代码 + 数据）
- 回滚时间窗口：30 分钟内完成
- 回滚验证：回滚后抽样 100 笔订单验证状态正确

**灰度发布策略**：
- 灰度范围：按用户 ID 尾号，先 1% -> 10% -> 50% -> 100%
- 灰度时长：每个阶段观察 3 天
- 灰度指标：
  - 订单创建成功率 > 99.9%
  - 状态流转成功率 > 99.9%
  - 客诉量无明显增加
- 紧急回滚：灰度期间发现问题，可立即关闭灰度开关

**风险提示**：
- 风险1：外部系统 B 未按时改造，导致接口下线延期
- 缓解措施：提前 1 个月通知，每周跟进改造进度
```

## 示例六：业务价值量化示例

### 需求点：自动审批功能

```markdown
【业务价值量化】

| 价值维度 | 当前状态 | 预期状态 | 量化指标 | 计算依据 |
|---------|---------|---------|---------|---------|
| 人力节省 | 人工审批，2人审批每单，平均耗时 30 分钟 | 系统自动审批，90% 订单自动通过 | 节省 1.8 人天/天 | 每日 100 单，90 单自动审批，节省 90×30÷60=45 小时/天 |
| 效率提升 | 审批周期平均 2 天 | 审批周期缩短至 2 小时 | 提升 96% | (2天-2小时)÷2天 = 96% |
| 错误减少 | 人工审批错误率约 3% | 自动审批错误率 < 0.1% | 减少 97% | (3%-0.1%)÷3% = 97% |
| 用户满意度 | 平均审批等待投诉 10 次/月 | 投诉降至 < 1 次/月 | 减少 90% | 投诉量下降 |
| 合规风险 | 人工审批无完整留痕 | 全程系统自动留痕 | 消除风险 | 审计日志完整 |

**价值说明**：
- 该功能为 Must have，因为涉及合规审计要求
- 预期年化节省人力成本：1.8 人天/天 × 250 工作日 × 1000 元/人天 = 45 万元/年
- 投资回收期：假设开发成本 20 万，回收期约 5 个月
```

## 示例七：苏格拉底式追问应用

### 功能点：库存扣减

```markdown
【苏格拉底式追问结果】

| 问题编号 | 提问角色 | 问题 | 风险等级 | 解决方案 |
|---------|---------|------|---------|---------|
| 1 | 后端工程师 | 当 1000 用户同时购买同一件商品时，如何保证不超卖？ | 高 | 使用分布式锁（Redis RedLock），锁粒度为商品 SKU |
| 2 | 后端工程师 | 如果库存扣减成功，但订单创建失败，如何回滚？ | 高 | 采用本地消息表方案，异步补偿回滚库存 |
| 3 | QA | 如何测试高并发场景下的库存一致性？ | 中 | 使用 JMeter 模拟 1000 并发，验证库存最终一致性 |
| 4 | 安全工程师 | 用户能否通过修改请求参数扣减其他商品的库存？ | 高 | 服务端校验商品 ID 与订单商品的绑定关系 |
| 5 | 产品经理 | 如果用户下单后 30 分钟未支付，库存如何处理？ | 中 | 设置订单超时时间为 30 分钟，超时自动释放库存 |

**结论**：
- 高风险问题 3 个，已在需求文档中补充相应处理逻辑
- 中风险问题 2 个，作为优化项后续处理
- 无需要用户确认的问题
```

## Pro 版快速模式

### 用户输入

```text

```

- 仍保留五个交付物

