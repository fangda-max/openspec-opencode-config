# Enterprise Requirement Doc Pro 使用文档

## 一、概述

`enterprise-requirement-doc-pro` 是一个**生产级别**的企业级需求文档生成技能，基于极端严谨的 PRD 标准，可以从混杂的原始材料（会议纪要、建设方案、功能说明书、原型截图等）自动生成结构化的**系统功能需求说明书**。

相比原版，Pro 版增加了 **AC 验收标准、Unhappy Path 深度分析、数据追踪埋点、版本兼容性分析、业务价值量化、苏格拉底式追问** 等生产级必备内容。

**核心特点：**
- 6 阶段工程化工作流 + Phase 3.5 深度分析
- 事实/推断/待确认三分法防幻觉
- 自动识别新增点、优化点、改造点
- **AC 验收标准**：Given-When-Then 格式，可测试可验证
- **Unhappy Path**：每个功能至少 2 个异常场景，具体到错误代码
- **数据追踪埋点**：核心功能强制要求埋点定义
- **版本兼容性分析**：改造点强制要求兼容性方案
- **业务价值量化**：Must have 需求必须量化价值
- **苏格拉底式追问**：防御性边界问题识别
- AI 自评报告 + AI 局限性声明
- 同时输出 Markdown + Word 文档

**Pro 版 vs 原版：**
| 特性 | 原版 | Pro 版 |
|------|------|--------|
| AC 验收标准 | 建议 | **强制要求** |
| Unhappy Path | 建议 | **强制要求（至少 2 个）** |
| 数据追踪埋点 | 无 | **核心功能强制要求** |
| 兼容性分析 | 简单提及 | **改造点强制要求** |
| 业务价值量化 | 建议 | **Must have 强制要求** |
| 苏格拉底式追问 | 无 | **新增** |
| AI 局限性声明 | 无 | **新增** |
| 交付物 | 4 个 | **5 个** |

## 二、使用前提

### 2.1 环境依赖

需要安装以下依赖（用于提取 docx、渲染 Word、PDF 处理等）：

```bash
pip install python-docx markdown pdfplumber pillow mammoth
```

### 2.2 原材料准备

你需要准备以下任意组合的**原材料**：

| 材料类型 | 格式 | 说明 |
|---------|------|------|
| 需求描述 | 文字 | 项目目标、边界、需求概述 |
| 会议纪要 | docx/pdf/md/txt | 背景、痛点、争议点 |
| 建设方案 | docx/pdf/md | 范围、场景、总体流程 |
| 功能说明书 | docx/pdf/md | 页面、按钮、规则、字段 |
| 原型/截图 | png/jpg/pdf | 页面布局、交互元素 |
| 历史需求文档 | docx/pdf/md | 术语、章节结构、迭代关系 |
| 公司模板 | docx/md | 章节命名、表格格式要求 |

### 2.3 原材料提供方式

**方式一：本地文件路径（推荐）**
```
请使用 Pro 版基于以下材料生成需求文档：
- D:\项目\会议纪要.docx
- D:\项目\建设方案.docx
- D:\项目\功能说明书.md
- D:\项目\公司模板.docx
```

**方式二：飞书云盘链接**
```
请使用 Pro 版基于以下材料生成需求文档：
- https://xxx.feishu.cn/docx/xxxxxx
- https://xxx.feishu.cn/docx/yyyyyy
```

**方式三：直接提供文件内容**
- 将文档内容直接粘贴到对话中

> ⚠️ 飞书消息中发送的文件附件（file_key 格式）无法直接读取，请提供**本地路径**或**云盘分享链接**。

## 四、完整工作流程

当你触发需求文档生成后，我会按以下 **6 个阶段 + Phase 3.5 深度分析**顺序执行：

```
Phase 1: Ingest Materials（读取材料）
    ↓
Phase 2: Build Fact Base（构建事实底稿）
    ↓
Phase 3: Generate Delta（生成中间稿：新增/优化/改造点）
    ↓
Phase 3.5: Deep Analysis（深度分析 - Pro 版核心）
    ├── 3.5.1 追问澄清（业务价值量化）
    ├── 3.5.2 场景拆解（主流程 + Unhappy Path + 边界）
    ├── 3.5.3 冲突检测
    ├── 3.5.4 数据流分析
    ├── 3.5.5 风险预判
    ├── 3.5.6 优先级排序（MoSCoW）
    └── 3.5.7 苏格拉底式追问（Pro 版新增）
    ↓
Phase 4: Draft Formal Doc（起草正式需求文档）
    ├── 必须包含 AC（Given-When-Then 格式）
    ├── 必须包含 Unhappy Path（至少 2 个）
    ├── 必须包含数据追踪埋点（核心功能）
    └── 必须包含兼容性分析（改造点）
    ↓
Phase 5: Produce MD & DOCX（输出文件）
    ↓
Phase 6: Self Review（AI 自评 - Pro 版增强维度）
    ├── 基础维度检查
    ├── Pro 版维度检查（AC、埋点、兼容性）
    └── 生成 AI 局限性声明
```

### Phase 1：读取材料
- 自动识别材料类型（docx/pdf/md/txt/图片）
- 使用脚本提取 docx 中的正文和表格
- 记录材料之间的冲突点

### Phase 2：构建事实底稿
- 从材料中抽取：背景、目标、范围、角色、功能、规则、接口等
- 每条信息标注来源材料
- 检查关键缺口（如缺失目标、范围、权限等）
- **若缺失关键信息，会先向你提问，不会凭空编造**

### Phase 3：生成中间稿
- 将需求分类为：**新增点** / **优化点** / **改造点**
- 每个点标注：来源、依据、影响范围、规则、**MoSCoW 优先级**
- **每个 Must have 点必须量化业务价值**
- 标记不确定事项为"待确认"

### Phase 3.5：深度分析（Pro 版核心）
- **业务价值追问**：必须用数据量化（节省 XX 人天/提升 XX% 效率）
- **场景拆解**：主流程 + **至少 2 个 Unhappy Path** + 边界 case
- **冲突检测**：原材料之间、新需求与历史需求、功能依赖
- **数据流分析**：数据流入、处理、流出、存储
- **风险预判**：技术/业务/合规/依赖风险及缓解措施
- **优先级排序**：MoSCoW 方法（Must/Should/Could/Won't）
- **苏格拉底式追问**：后端工程师/QA/安全工程师视角的防御性问题

### Phase 4：起草正式文档（Pro 版增强）
- 判断文档规模（轻量/标准/完整）
- 判断能力类型（A类传统功能/B类AI异步功能）
- 融合公司模板与默认模板
- 逐章节生成正式需求说明书
- **必须包含**：
  - **AC 验收标准**：Given-When-Then 格式，至少 3 个
  - **Unhappy Path**：至少 2 个，具体到错误代码
  - **数据追踪埋点**：核心功能必须定义
  - **兼容性分析**：改造点必须分析

### Phase 5：输出文件
- 生成 Markdown 版本
- 渲染 Word (.docx) 版本

### Phase 6：AI 自评（Pro 版增强）
- 检查准确性、完整性、可行性、一致性
- **Pro 版新增检查维度**：
  - AC 完整性（覆盖率、格式正确性）
  - Unhappy Path 充分性（数量、具体性）
  - 数据追踪埋点完整性
  - 兼容性分析完整性
  - 业务价值量化完整性
- 若发现高风险问题，先修正文档再交付
- 输出自评报告
- 输出 AI 局限性声明

## 七、常见问题

### Q：Pro 版与原版有什么区别？
**A：** Pro 版增加了 AC 验收标准、Unhappy Path 深度分析、数据追踪埋点、版本兼容性分析、业务价值量化、苏格拉底式追问等生产级必备内容。原版继续保留，适合常规需求。

### Q：支持哪些文件格式？
**A：** docx、pdf、md、txt、png、jpg。不支持 `.doc` 格式，请先用 Word 另存为 `.docx`。

### Q：飞书文件怎么提供？
**A：** 两种方式：
1. 将文件下载到本地，提供文件路径
2. 在飞书云盘中创建分享链接，提供链接

### Q：材料不充分怎么办？
**A：** 我会在 Phase 2 检查关键缺口，主动向你提问补充。不会凭空编造。

### Q：生成的文档不准确怎么办？
**A：** 查看 `03-AI自评报告.md` 中的问题清单，告诉我需要修正的部分，我会重新生成。

### Q：怎么确认文档质量？
**A：** 查看自评报告，重点关注：
- 是否有"高风险"项
- 是否有"待确认"项需要你补充
- **Pro 版新增**：AC 完整性、埋点完整性、兼容性分析是否通过
- 各维度评分是否为"通过"

### Q：AI 生成的文档可以直接用吗？
**A：** 详见 `04-AI局限性声明.md`。AI 生成的文档是**初稿**，必须经过人工复核后方可使用。特别是业务规则、性能指标、权限设计、兼容性方案等需要重点复核。

### Q：为什么 AC 必须采用 Given-When-Then 格式？
**A：** GWT 格式是国际通用的 AC 写作标准，具有以下优势：
- 清晰明确：前置条件、操作、预期结果一目了然
- 可测试：每个 AC 都可以转化为测试用例
- 无歧义：避免"支持XX功能"这类模糊描述
- 便于评审：开发、测试、业务方都能快速理解

### Q：为什么改造点必须写兼容性分析？
**A：** 改造项目往往涉及历史数据、接口变更、流程调整，如果没有兼容性分析，容易导致：
- 历史数据无法迁移，导致数据丢失
- 接口变更影响下游系统，导致集成失败
- 新功能上线后无法回滚，导致线上事故
兼容性分析是改造项目的"保险绳"。

## 九、Pro 版最佳实践

### 9.1 何时使用 Pro 版？

**推荐使用 Pro 版：**
- 需要通过严格评审的 PRD（政府、金融、大型企业）
- 涉及改造的存量系统升级
- 核心业务系统需求
- 有明确的测试验收要求
- 需要数据追踪分析的功能

**原版即可满足：**
- 内部工具系统
- 快速迭代的敏捷项目
- 概念验证（POC）阶段
- 对文档深度要求不高的项目

### 9.2 如何高效使用 Pro 版？

1. **材料准备充分**：提供的材料越多，生成的文档质量越高
2. **明确公司模板**：如果有公司模板，请一并提供
3. **关注中间稿**：Phase 3 生成的中间稿是审阅的关键节点
4. **积极补充信息**：对于【待确认】的事项，及时补充
5. **人工复核必不可少**：即使自评通过，也要按照 `04-AI局限性声明.md` 复核

### 9.3 Pro 版常见误区

❌ **误区1**：AI 生成的文档可以直接交付开发  
✅ **正确**：AI 生成的是初稿，必须经过人工复核

❌ **误区2**：提供了材料就不需要回答了  
✅ **正确**：Pro 版会追问关键缺口，请积极配合

❌ **误区3**：快速模式等于低质量  
✅ **正确**：快速模式只是减少确认轮次，质量标准不变

❌ **误区4**：AC 和 Unhappy Path 是"形式主义"  
✅ **正确**：这是生产级 PRD 的标配，能显著减少后期返工
