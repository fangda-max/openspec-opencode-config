# 写作规范与转化规则

本文件定义正式需求文档的写法，目标是让输出结果具备企业级可读性、可评审性、可落地性。

## 1. 总体原则

- 先业务、后页面、再规则、最后指标。
- 先全局、后局部；先流程、后字段。
- 同一概念全文使用同一术语。
- 一切结论以原材料为准，不得凭空补规则。
- 对不确定内容，明确标注"待确认"或"建议值"。

## 2. 原材料到章节的映射

| 原材料 | 优先转化章节 | 常见提取点 |
|------|------|------|
| 会议纪要 | 背景、痛点、争议点 | 现状问题、风险、目标诉求 |
| 建设方案 | 范围边界、总体流程 | 场景、建设内容、能力层次 |
| 功能说明书 | 系统功能 | 页面、按钮、字段、状态、规则 |
| 界面原型/截图 | 系统功能 | 页面布局、交互元素、文案 |
| 历史需求文档 | 术语、结构、集成上下文 | 章节命名、对接对象、沿用逻辑 |
| 公司模板 | 全文结构和格式 | 章节、表格、命名规范 |

## 3. 用词规范

### 默认推荐词

| 概念 | 默认用词 |
|------|------|
| 增 | 新增 |
| 改 | 编辑 |
| 查 | 查看 |
| 删 | 删除 |
| 启用 | 生效 |
| 停用 | 失效 |
| 数据项 | 字段 |
| 对话层 | 弹窗 |
| 侧边浮层 | 抽屉 |

### 使用要求
- 首次出现的角色，给出完整名称。
- 状态值使用统一写法，如 `"待提交"`、`"已生效"`。
- 按钮文案统一用 `【】` 包裹。

## 4. 句式规范

### 适合写进正式文档的句式
- `系统提供对 XXX 的统一查询与维护能力。`
- `用户点击【查询】后，系统按条件发起查询请求。`
- `若校验未通过，则提示 XXX，且不允许提交。`
- `当对象状态为 "待审核" 时，仅审核员可执行【通过】与【驳回】。`

### 不适合写进正式文档的句式
- `这里就是一个列表页。`
- `用户点一下就行。`
- `应该差不多是这样。`

## 5. 章节写法

### 5.1 背景
- 第一段写业务处境。
- 第二段写现状问题，建议 3-5 条。
- 第三段写建设必要性。

### 5.2 建设目标
- 用业务目标，不用技术路线替代目标。
- 每个目标一句话，最好能描述业务收益。

### 5.3 范围与边界
- 范围按层次写，边界按"包含/不包含"写。
- 线下流程、第三方能力、历史系统共存关系要说明清楚。

### 5.4 总体业务流程
- 描述触发人、触发动作、核心节点、终点。
- 如果流程之间存在反哺和闭环，要专门写一节。

### 5.5 系统功能
- 页面详情必须能回答：
  - 从哪里进入
  - 谁能用
  - 页面长什么样
  - 按钮怎么工作
  - 有什么校验
  - 状态如何变化
  - 字段是什么意思

## 6. 页面/操作扩写规则

### 从"功能点"扩成"功能章节"

如果原材料只给出一句功能点：

```text
支持待生效记录执行生效、编辑、删除
```

正式文档需要扩成：
- 谁能看到这些按钮
- 这些按钮出现在哪个页面
- 各按钮的校验规则是什么
- 操作后的状态变化是什么
- 是否需要二次确认
- 成功/失败如何反馈

### 从"界面截图"扩成"界面原型描述"

按以下顺序描述：
1. 页面形态：页面/弹窗/抽屉
2. 上中下结构或左右结构
3. 查询区、列表区、表单区、结果区
4. 按钮、字段、表格列、提示文案

## 7. A/B 类功能写法

### A 类
适用于 CRUD、查询、审批、台账、基础配置、普通流程节点。

必须写清：
- 操作入口
- 功能描述
- 使用角色
- 界面原型
- 实现逻辑
- 字段说明

### B 类
适用于 AI 识别、推荐、生成、异步任务、定时任务、批处理。

除 A 类内容外，再补：
- 交互过程
- 输入输出约束
- 提示词或规则框架
- 知识来源/语料来源
- 异步机制
- 效果指标

## 8. 缺口补全策略

### 可用建议值
- 页面首屏：`<= 2 秒`
- 核心操作：`<= 3 秒`
- 复杂查询：`<= 5 秒`
- 同步接口：`95% <= 500ms`
- 审计日志留存：`>= 6 个月`

### 必须提问确认
- 核心对象有哪些状态
- 各角色的权限边界
- 本期不做什么
- 要对接哪些系统
- 哪些字段有特殊规则

## 9. 正式成文前自查

- 是否已经区分事实和推断
- 是否已经补齐角色、状态、范围、流程
- 是否所有页面都写清入口和按钮逻辑
- 是否所有字段都能说明业务含义
- 是否所有异常场景都有处理方式
- 是否仍存在明显空泛描述
