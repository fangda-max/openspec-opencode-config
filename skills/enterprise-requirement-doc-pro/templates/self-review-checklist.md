# AI 自评清单（Pro 版增强）

本文件用于生成 `03-AI自评报告.md`，并决定正式需求文档是否可以直接交付评审。

**Pro 版增强**：本清单增加了 AC 完整性、数据追踪埋点、兼容性分析、苏格拉底式追问等检查项。

## 1. 自评目标

自评必须回答四个问题：
- 是否准确反映原材料
- 是否完整覆盖核心需求
- 是否具备实施可行性
- 是否存在明显偏离、冲突或幻觉扩写

**Pro 版新增**：
- AC 验收标准是否完整（Given-When-Then 格式）
- Unhappy Path 是否充分（至少 2 个）
- 数据追踪埋点是否定义
- 兼容性分析是否完整（改造点）

## 2. 评分维度

每个维度按 `通过 / 有风险 / 不通过` 判断，并给出证据。

| 维度 | 检查点 |
|------|------|
| 准确性 | 是否与原材料事实一致，是否误读 |
| 完整性 | 目标、范围、角色、流程、规则、字段是否缺失 |
| 可行性 | 权限、状态、接口、流程是否可落地 |
| 一致性 | 术语、角色、状态、字段、按钮名称是否统一 |
| 不偏离原材料 | 是否加入原材料没有支撑的关键结论 |
| **深度充分性** | **是否经过深度分析，内容是否流于表面** |
| **AC 完整性（Pro 新增）** | **是否每个功能点都有 Given-When-Then 格式的 AC** |
| **Unhappy Path 充分性（Pro 新增）** | **是否每个功能点都有至少 2 个 Unhappy Path** |
| **数据追踪完整性（Pro 新增）** | **核心功能是否都有埋点定义** |
| **兼容性分析（Pro 新增）** | **改造点是否都有兼容性分析** |

## 3. 高风险打回条件

出现以下任一情况，应先回改文档，再交付：
- 编造关键业务规则
- 编造接口、字段、状态值、角色权限
- 将待确认事项写成确定结论
- 不同章节存在明显冲突
- 流程无法闭环
- 关键功能模块缺失
- **Pro 版新增**：
  - Must have 需求缺少业务价值量化
  - 核心功能点缺少 AC 验收标准
  - 改造点缺少兼容性分析
  - Unhappy Path 描述模糊，无法执行

## 4. 自评执行步骤

### 第一步：原材料对照
- 对照原材料，检查正式文档中每个核心章节是否有来源支撑。
- 对"范围、边界、状态流转、角色权限、关键规则"重点复核。

### 第二步：结构检查
- 是否匹配轻量/标准/完整结构
- 是否所有复杂模块都写到页面/操作层
- 是否包含必要的非功能、安全、权限内容
- **Pro 版新增**：
  - 是否包含数据追踪埋点章节
  - 改造点是否包含兼容性分析章节

### 第三步：逻辑检查
- 状态流转是否闭环
- 按钮逻辑是否与状态和角色匹配
- 数据来源、流向是否说得清楚
- 异常处理是否覆盖关键分支
- **Pro 版新增**：
  - AC 是否可测试、可验证
  - Unhappy Path 是否有具体的错误代码和用户提示
  - 埋点事件是否与功能点对应

### 第四步：偏离检查
- 是否存在"材料未提、却被写死"的要求
- 是否把建议值误写成硬性结论
- 是否把技术方案误写成业务目标

### 第五步：深度充分性检查（Pro 版增强）⚡

基于 Phase 3.5 Deep Analysis 的输出，检查：

#### 基础深度检查（原）

- [ ] **业务价值**：每个功能都有明确的业务价值说明，且能量化
- [ ] **目标用户**：每个功能都有明确的目标用户和使用场景
- [ ] **场景覆盖**：主流程、异常流、边界 case 都已覆盖
  - [ ] 正常流程是否完整？
  - [ ] 异常情况是否有处理逻辑？
  - [ ] 边界 case 是否考虑？（空值、超大值、特殊字符、高并发等）
- [ ] **冲突检测**：需求之间无逻辑冲突，数据定义一致
- [ ] **数据流**：数据如何流入、处理、流出、存储，流转清晰
- [ ] **风险识别**：技术/业务/合规风险已识别，有缓解措施
- [ ] **优先级**：需求已按 MoSCoW 方法排序，明确 MVP 范围

#### Pro 版新增检查项

- [ ] **AC 完整性**
  - [ ] 每个功能点是否都有至少 3 个 AC？
  - [ ] AC 是否采用 Given-When-Then 格式？
  - [ ] AC 是否覆盖主流程、业务异常、系统异常三种场景？
  - [ ] AC 是否可测试、可验证？
  - [ ] AC 是否使用具体数据而非模糊描述？

- [ ] **Unhappy Path 充分性**
  - [ ] 每个功能点是否都有至少 2 个 Unhappy Path？
  - [ ] Unhappy Path 是否区分为业务异常和系统异常？
  - [ ] 每个 Unhappy Path 是否有明确的触发条件？
  - [ ] 每个 Unhappy Path 是否有具体的错误代码？
  - [ ] 每个 Unhappy Path 是否有用户提示文案？
  - [ ] 每个 Unhappy Path 是否有系统处理逻辑？

- [ ] **数据追踪埋点完整性**
  - [ ] 核心功能（增删改查、关键操作）是否都有埋点？
  - [ ] 埋点事件列表是否完整（事件名称、触发时机、关键属性）？
  - [ ] 事件属性是否包含通用属性（user_id, timestamp, session_id）？
  - [ ] 敏感数据是否明确标注需要脱敏？
  - [ ] 埋点数据保留期限是否明确？

- [ ] **兼容性分析完整性（仅改造点）**
  - [ ] 每个改造点是否都有兼容性分析？
  - [ ] 是否包含历史数据迁移方案？
  - [ ] 是否包含接口兼容性分析？
  - [ ] 是否包含功能回滚方案？
  - [ ] 是否包含灰度发布策略？

- [ ] **业务价值量化**
  - [ ] 每个 Must have 需求是否都有量化的业务价值？
  - [ ] 业务价值是否包含具体数值（节省XX人天/提升XX%效率）？
  - [ ] 业务价值的计算依据是否说明？

- [ ] **苏格拉底式追问**
  - [ ] 是否已执行苏格拉底式追问（后端工程师/QA/安全工程师视角）？
  - [ ] 是否识别了高风险问题？
  - [ ] 高风险问题是否已在文档中体现或标注【待确认】？

#### 深度不足的警示信号（Pro 版增强）

- 功能描述只有一句话，没有展开
- 只有正常流程，没有异常处理
- 字段只有名称，没有类型/长度/校验规则
- 使用了"待定""可能""正常情况下"等模糊词汇
- 没有量化指标（使用率、响应时间、并发量等）
- **AC 使用"支持XX功能"的功能描述**
- **Unhappy Path 只写"异常情况系统提示"而不具体说明**
- **改造点不写兼容性分析**
- **埋点只有事件名称，没有属性定义**

## 6. 自评输出要求

- 问题优先，不写空泛好评。
- 每个问题尽量给出证据来源。
- 不要把"可优化建议"伪装成"严重缺陷"。
- 若未发现明显问题，也要写残余风险和测试盲区。
- **Pro 版新增**：
  - 必须量化各维度覆盖率
  - 必须列出具体的 AC/UP/埋点问题清单
  - 必须提示 AI 局限性内容

## 7. 最终放行标准（Pro 版增强）

仅当满足以下条件时，才建议"可直接提交评审"：

### 基础标准
- 无高风险打回项
- 关键章节齐全
- 核心流程闭环
- 角色、状态、字段、按钮名称统一
- 与原材料无重大偏离
- 深度分析充分（业务价值明确、场景覆盖完整、风险已识别）

### Pro 版增强标准
- **AC 完整性**：核心功能 AC 覆盖率 100%，格式正确
- **Unhappy Path 充分性**：每个功能点至少 2 个 UP，描述具体
- **数据追踪完整性**：核心功能埋点定义完整
- **兼容性分析**：改造点 100% 包含兼容性分析
- **业务价值量化**：Must have 需求 100% 量化
- **AI 局限性声明**：已生成 `04-AI局限性声明.md`

**注意**：即使基础维度通过，Pro 版增强维度不通过也不建议直接提交评审。
