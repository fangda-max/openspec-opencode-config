# Enterprise Requirement Doc 使用文档

## 一、概述

`enterprise-requirement-doc` 是一个企业级需求文档生成技能，可以从混杂的原始材料（会议纪要、建设方案、功能说明书、原型截图等）自动生成结构化的**系统功能需求说明书**。

**核心特点：**
- 6 阶段工程化工作流
- 事实/推断/待确认三分法防幻觉
- 自动识别新增点、优化点、改造点
- AI 自评报告确保文档质量
- 同时输出 Markdown + Word 文档

---

## 二、使用前提

### 2.1 环境依赖

需要安装 `python-docx`（用于提取 docx 和渲染 Word 文档）：

```bash
pip install python-docx
```

### 2.2 原材料准备

你需要准备以下任意组合的**原材料**：

| 材料类型 | 格式 | 说明 |
|---------|------|------|
| 需求描述 | 文字 | 项目目标、边界、需求概述 |
| 会议纪要 | docx/pdf/md/txt | 背景、痛点、争议点 |
| 建设方案 | docx/pdf/md | 范围、场景、总体流程 |
| 功能说明书 | docx/pdf/md | 页面、按钮、规则、字段 |
| 原型/截图 | png/jpg/pdf | 页面布局、交互元素 |
| 历史需求文档 | docx/pdf/md | 术语、章节结构、迭代关系 |
| 公司模板 | docx/md | 章节命名、表格格式要求 |

### 2.3 原材料提供方式

**方式一：本地文件路径（推荐）**
```
请基于以下材料生成需求文档：
- D:\项目\会议纪要.docx
- D:\项目\建设方案.docx
- D:\项目\功能说明书.md
- D:\项目\公司模板.docx
```

**方式二：飞书云盘链接**
```
请基于以下材料生成需求文档：
- https://xxx.feishu.cn/docx/xxxxxx
- https://xxx.feishu.cn/docx/yyyyyy
```

**方式三：直接提供文件内容**
- 将文档内容直接粘贴到对话中

> ⚠️ 飞书消息中发送的文件附件（file_key 格式）无法直接读取，请提供**本地路径**或**云盘分享链接**。

## 四、完整工作流程

当你触发需求文档生成后，我会按以下 **6 个阶段**顺序执行：

```
Phase 1: Ingest Materials（读取材料）
    ↓
Phase 2: Build Fact Base（构建事实底稿）
    ↓
Phase 3: Generate Delta（生成中间稿：新增/优化/改造点）
    ↓
Phase 4: Draft Formal Doc（起草正式需求文档）
    ↓
Phase 5: Produce MD & DOCX（输出文件）
    ↓
Phase 6: Self Review（AI 自评）
```

### Phase 1：读取材料
- 自动识别材料类型（docx/pdf/md/txt/图片）
- 使用脚本提取 docx 中的正文和表格
- 记录材料之间的冲突点

### Phase 2：构建事实底稿
- 从材料中抽取：背景、目标、范围、角色、功能、规则、接口等
- 每条信息标注来源材料
- 检查关键缺口（如缺失目标、范围、权限等）
- **若缺失关键信息，会先向你提问，不会凭空编造**

### Phase 3：生成中间稿
- 将需求分类为：**新增点** / **优化点** / **改造点**
- 每个点标注：来源、依据、影响范围、规则、优先级
- 标记不确定事项为"待确认"

### Phase 4：起草正式文档
- 判断文档规模（轻量/标准/完整）
- 判断能力类型（A类传统功能/B类AI异步功能）
- 融合公司模板与默认模板
- 逐章节生成正式需求说明书

### Phase 5：输出文件
- 生成 Markdown 版本
- 渲染 Word (.docx) 版本

### Phase 6：AI 自评
- 检查准确性、完整性、可行性、一致性
- 若发现高风险问题，先修正文档再交付
- 输出自评报告

## 六、进阶用法

### 6.1 快速模式

如果材料已经很充分，不想中间反复确认：

```
请用快速模式生成需求文档，材料在 D:\项目\ 下
```

快速模式会减少中间确认轮次，但缺失关键信息时仍会提问。

### 6.2 自定义文件命名

```
请生成需求文档，命名为"XX系统需求说明书V2.0"
```

### 6.3 指定文档规模

```
请生成轻量版需求文档（项目只有3个模块）
```

可选：
- **轻量**：<=3 个功能模块
- **标准**：4-10 个功能模块
- **完整**：>10 个模块或复杂系统

### 6.4 仅生成中间稿

```
请先帮我整理需求的新增/优化/改造点，不需要生成正式文档
```

### 6.5 基于历史文档迭代

```
请基于 D:\项目\历史需求V1.0.docx 的基础上，
结合 D:\项目\新需求V2.0.docx 生成迭代版需求文档
```

## 七、常见问题

### Q：支持哪些文件格式？
**A：** docx、pdf、md、txt、png、jpg。不支持 `.doc` 格式，请先用 Word 另存为 `.docx`。

### Q：飞书文件怎么提供？
**A：** 两种方式：
1. 将文件下载到本地，提供文件路径
2. 在飞书云盘中创建分享链接，提供链接

### Q：材料不充分怎么办？
**A：** 我会在 Phase 2 检查关键缺口，主动向你提问补充。不会凭空编造。

### Q：生成的文档不准确怎么办？
**A：** 查看 `03-AI自评报告.md` 中的问题清单，告诉我需要修正的部分，我会重新生成。

### Q：怎么确认文档质量？
**A：** 查看自评报告，重点关注：
- 是否有"高风险"项
- 是否有"待确认"项需要你补充
- 各维度评分是否为"通过"

> **你：** 请基于 D:\Desktop\运营中心需求文档\ 目录下的所有材料，生成一份系统功能需求说明书，并使用快速模式。

### 示例 3：迭代升级

> **你：** 请基于历史文档 D:\项目\需求V1.0.docx，结合新需求材料 D:\项目\新需求.docx，生成 V2.0 需求说明书。
>
> **我会：** 对比新旧材料 → 识别新增/优化/改造点 → 基于V1.0结构生成V2.0 → 标注迭代变化。
