# 系统功能需求说明书模板

本模板用于生成 `02-系统功能需求说明书.md`。如用户提供公司模板，以公司模板章节为主；本模板作为默认补充结构。

## 1. 规模选择

| 规模 | 适用条件 | 推荐结构 |
|------|------|------|
| 轻量 | <=3 个功能模块，集成少 | 3 章 |
| 标准 | 4-10 个功能模块 | 7 章 |
| 完整 | 多系统集成、角色复杂、AI/异步能力明显 | 7 章 + 附录 |

## 4. 各章节写作要点

### 一、背景与建设目标
- 说明业务背景、现状问题、建设必要性。
- 建设目标建议 3-5 条，尽量体现业务价值而非技术实现。
- 有内部概念或行业术语时补充术语表。

### 二、范围与边界
- 范围建议按"基础层/核心层/支撑层"分层描述。
- 边界必须同时写"包含"和"不包含"。
- 若存在外部依赖、线下流程、历史系统共存，也在本章说明。

### 三、总体业务流程概览
- 以业务语言描述主流程，不要直接写接口流程。
- 每个流程至少说明"触发起点、核心环节、终点"。
- 如存在数据回流、反哺、闭环，要单独写一节。

### 四、系统功能
- 这是正文核心。
- 简单模块可写"业务场景与流程 + 功能详情"。
- 复杂模块必须写"业务目标 + 业务场景 + 业务流程 + 页面/操作详情"。
- 页面/操作详情统一使用下方模板。

### 五、非功能需求
- 默认至少覆盖性能、安全、审计三类。
- 缺少具体指标时，可使用企业级通用占位值，但要注明"建议值"或"待确认"。

### 六、系统接口
- 无集成可省略。
- 有集成但信息不完整时，先列接口清单并标"待研发梳理"。

### 七、权限管理
- 角色复杂时单列。
- 若权限简单，可合并入各功能模块，但至少在本章给出角色矩阵摘要。

## 5. 功能模块详情模板

### A 类功能模板

```markdown
##### X.X.X 页面/操作名称

【操作入口】：菜单路径、父页面按钮或流程节点

【功能描述】：一句话说明该功能解决的问题

【使用角色】：角色A、角色B

【界面原型】：
- 查询区：字段、筛选条件、按钮
- 列表区：表头、操作列、状态展示
- 表单区：分组、字段、上传区、说明区
- 底部操作区：提交、保存、关闭、返回

【实现逻辑】：
- 页面初始化：进入页面时系统默认执行什么
- 查询逻辑：条件组装、校验、默认排序、分页
- 按钮逻辑：按钮权限、点击动作、校验规则、处理结果
- 状态流转：状态A -> 触发条件 -> 状态B
- 异常处理：校验失败、权限不足、无数据、执行失败时的反馈

【字段说明】：
| 字段名 | 类型 | 必填 | 可编辑 | 说明 |
|------|------|------|------|------|
| XXX | 文本 | 是 | 否 | XXX |
```

### B 类功能补充模板

在 A 类基础上，按需补充：

```markdown
【交互过程】：
1. 正在识别/分析/处理...
2. 正在提取/计算/比对...
3. 处理完成

【提示词(Prompt)】：
- 角色定位
- 输入约束
- 输出格式

【语料/知识库】：
- 来源1：XXX
- 来源2：XXX

【异步/批处理机制】：
- 触发条件：XXX
- 执行策略：手动/事件/定时
- 并发与重试：XXX
- 结果通知：站内消息/轮询/回调

【验证效果】：
- 准确率/召回率/成功率/吞吐量/平均耗时
```

## 7. 附录建议

以下情况建议补附录：

- 关键对象较多
- 状态机复杂
- 接口字段较多
- 参考材料较多
- 用户要求完整交付版文档
