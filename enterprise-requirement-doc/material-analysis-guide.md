# 原材料解析指南

本文件用于把混杂的原始材料，稳定转化为可写需求文档的事实底稿。

## 1. 材料分类

| 材料类型 | 常见来源 | 主要贡献 |
|------|------|------|
| 需求描述 | 用户自然语言、IM消息 | 项目目标、边界、紧急程度 |
| 会议纪要/录音稿 | 会议记录、访谈稿 | 背景、痛点、现状、争议点 |
| 建设方案 | 方案文档、汇报稿 | 范围、场景、总体流程、建设思路 |
| 功能说明书 | 功能清单、开发说明 | 页面、按钮、规则、字段、状态 |
| 原型/截图 | PDF、图片、Axure导出 | 页面区域、交互、展示项 |
| 既有需求文档 | 历史版本、同类项目 | 术语、章节结构、集成上下文 |
| 模板规范 | 公司模板、交付规范 | 章节命名、表格结构、格式要求 |
| 技术约束材料 | 接口说明、架构说明 | 集成方式、数据来源、限制条件 |

## 2. 事实底稿抽取维度

每次分析至少按下表输出一轮结构化提取。

| 维度 | 需要抽取的内容 | 典型来源 |
|------|------|------|
| 业务背景 | 业务场景、现状、痛点、风险 | 会议纪要、建设方案 |
| 建设目标 | 为什么做、预期效果、价值 | 需求描述、方案、汇报材料 |
| 范围 | 本期做什么 | 方案、模板、历史需求 |
| 边界 | 本期不做什么、依赖谁 | 方案、会议纪要 |
| 角色职责 | 谁发起、谁审批、谁维护、谁查看 | 需求描述、功能说明 |
| 核心对象 | 主体对象、属性、状态、主键 | 功能说明、表格、接口说明 |
| 状态流转 | 状态、触发条件、前后置约束 | 功能说明、流程图、会议纪要 |
| 功能模块 | 模块清单、子功能清单 | 方案、功能说明、既有需求 |
| 页面交互 | 入口、区域、字段、按钮、跳转 | 原型、截图、功能说明 |
| 业务规则 | 唯一性、校验、权限、时限、阈值 | 功能说明、会议纪要 |
| 接口与数据 | 对接系统、入参、出参、数据来源 | 架构说明、接口说明 |
| 非功能 | 性能、安全、审计、可用性 | 模板、方案、企业规范 |
| 风险与待确认 | 冲突点、缺口、假设 | 所有材料交叉分析 |

## 3. 三分法标注

每条信息必须标注为以下三类之一：

| 类型 | 定义 | 使用方式 |
|------|------|------|
| 事实 | 材料中直接出现，或多材料一致支撑 | 可直接进入正式文档 |
| 推断 | 根据多处材料合理归纳，但未被单点明确写出 | 仅可在中间稿或待确认区使用，正式文档需谨慎 |
| 待确认 | 关键材料缺失、材料冲突、无法稳定推断 | 先提问，未确认前不得写死 |

推荐标记格式：

```markdown
- 事实：项目经理可在【进度填报】中发起印章稽核。
  - 来源：功能说明书、界面截图

- 推断：印章稽核结果保存后应记录操作日志。
  - 依据：已有页面存在“保存结果”动作，且同系统对关键操作有留痕要求

- 待确认：异常结果是否拦截主流程流转。
  - 原因：材料中同时出现“提示风险”和“继续流转”两种表达
```

## 4. 新增点/优化点/改造点判定规则

中间稿必须区分三类需求变化。

| 类型 | 判定标准 | 典型表达 |
|------|------|------|
| 新增点 | 原系统无该能力，需要全新建设 | 新增模块、新增流程、新增页面、新增接口 |
| 优化点 | 原能力存在，但体验、效率、准确性、规则需加强 | 优化页面布局、补全校验、增强检索、提升效率 |
| 改造点 | 现有流程/页面/规则/数据结构需重构或替换 | 改造字段、替换流程、重做逻辑、兼容旧能力 |

判定优先级：
1. 若材料明确写“新增/新建/建设”，优先判定为新增点。
2. 若材料写“优化/增强/补充/完善”，优先判定为优化点。
3. 若材料写“改造/调整/替换/重构/迁移/升级现有”，优先判定为改造点。
4. 若无法判定，放入待确认，不要擅自分类。

## 5. 冲突处理规则

若原材料之间存在冲突，按以下顺序处理：

1. 用户口头明确说明
2. 最新版公司模板或正式发文
3. 最新版功能说明书/需求文档
4. 建设方案
5. 会议纪要/访谈稿
6. 你自己的推断

冲突记录格式：

```markdown
## 冲突项
- 冲突主题：是否拦截主流程
- 材料A：会议纪要表述为“提示风险后继续”
- 材料B：方案文档表述为“风险拦截”
- 当前处理：列为待确认，不写入正式文档定论
```

## 6. 缺口判断

### 必须补齐
- 至少一个明确的建设目标
- 功能模块清单
- 核心对象及状态/流程

### 强烈建议补齐
- 用户角色及权限
- 页面入口与关键字段
- 范围边界
- 集成对象

### 可先占位
- 接口详细字段
- 非功能具体数值
- 审计日志留存时长

## 7. 中间事实底稿建议结构

```markdown
# 事实底稿

## 1. 项目概况
## 2. 背景与痛点
## 3. 建设目标
## 4. 范围与边界
## 5. 角色与职责
## 6. 核心对象与状态
## 7. 功能模块清单
## 8. 页面与交互
## 9. 业务规则
## 10. 接口与数据
## 11. 非功能需求
## 12. 风险、冲突与待确认
```

## 8. 写正式文档前的检查

- 是否已列出全部材料来源
- 是否已标出冲突项
- 是否已区分事实/推断/待确认
- 是否已能回答“为什么做、做什么、谁来用、怎么流转、有哪些规则”
- 若不能，先补问，再写文档
