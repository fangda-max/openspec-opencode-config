# AI 自评清单

本文件用于生成 `03-AI自评报告.md`，并决定正式需求文档是否可以直接交付评审。

## 1. 自评目标

自评必须回答四个问题：
- 是否准确反映原材料
- 是否完整覆盖核心需求
- 是否具备实施可行性
- 是否存在明显偏离、冲突或幻觉扩写

## 2. 评分维度

每个维度按 `通过 / 有风险 / 不通过` 判断，并给出证据。

| 维度 | 检查点 |
|------|------|
| 准确性 | 是否与原材料事实一致，是否误读 |
| 完整性 | 目标、范围、角色、流程、规则、字段是否缺失 |
| 可行性 | 权限、状态、接口、流程是否可落地 |
| 一致性 | 术语、角色、状态、字段、按钮名称是否统一 |
| 不偏离原材料 | 是否加入原材料没有支撑的关键结论 |

## 3. 高风险打回条件

出现以下任一情况，应先回改文档，再交付：
- 编造关键业务规则
- 编造接口、字段、状态值、角色权限
- 将待确认事项写成确定结论
- 不同章节存在明显冲突
- 流程无法闭环
- 关键功能模块缺失

## 4. 自评执行步骤

### 第一步：原材料对照
- 对照原材料，检查正式文档中每个核心章节是否有来源支撑。
- 对“范围、边界、状态流转、角色权限、关键规则”重点复核。

### 第二步：结构检查
- 是否匹配轻量/标准/完整结构
- 是否所有复杂模块都写到页面/操作层
- 是否包含必要的非功能、安全、权限内容

### 第三步：逻辑检查
- 状态流转是否闭环
- 按钮逻辑是否与状态和角色匹配
- 数据来源、流向是否说得清楚
- 异常处理是否覆盖关键分支

### 第四步：偏离检查
- 是否存在“材料未提、却被写死”的要求
- 是否把建议值误写成硬性结论
- 是否把技术方案误写成业务目标

## 6. 自评输出要求

- 问题优先，不写空泛好评。
- 每个问题尽量给出证据来源。
- 不要把“可优化建议”伪装成“严重缺陷”。
- 若未发现明显问题，也要写残余风险和测试盲区。

## 7. 最终放行标准

仅当满足以下条件时，才建议“可直接提交评审”：
- 无高风险打回项
- 关键章节齐全
- 核心流程闭环
- 角色、状态、字段、按钮名称统一
- 与原材料无重大偏离
